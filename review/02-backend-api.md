# Refactoring Plan: Backend API / Service Layers

**Scope:** `computor-backend/src/computor_backend/` — `api/`, `business_logic/`, `services/`, `interfaces/`, `websocket/`, plus entrypoints (`server.py`, `src/server.py`, `src/cli.py`, `src/seeder.py`)
**Reviewed:** 2026-07-07 (branch `release/2026.10`)
**Total LOC in scope:** ~38,100 (api/ 13,801 · business_logic/ 15,206 · services/ 2,892 · interfaces/ 2,689 · websocket/ 2,550)

## Architecture context (read first)

Intended layering: `api/` (routers) → `business_logic/` (domain orchestration, raises typed `ComputorException` subclasses of `HTTPException`) → `repositories/` + `model/`; `services/` holds infra adapters (MinIO storage, git, Redis-backed tutor-test state); `interfaces/` binds computor-types DTOs to SQLAlchemy models and powers a generic `CrudRouter`/`LookUpRouter` (`api/api_builder.py`) that auto-generates full CRUD for ~17 entities. The pattern is genuinely followed for core entities. It breaks down in the hand-rolled routers: `api/submissions.py`, `api/tests.py`, `api/examples.py`, `api/system.py`, and parts of `api/tutor.py` run queries, permission checks, commits, and Temporal orchestration inline — some of it a verbatim copy of functions that already exist in `business_logic/`. No layering-direction violations (nothing below `api/` imports `api/`).

**Key metrics:** 253 decorator-declared endpoints + ~95–120 generated by 17 `CrudRouter` / 5 `LookUpRouter` registrations. Largest files: `api/examples.py` 1434, `business_logic/messages.py` 1336, `api/submissions.py` 1301, `business_logic/course_git.py` 1195, `business_logic/lecturer_deployment.py` 1102, `api/tutor.py` 1099, `business_logic/users.py` 1058. 167 broad `except Exception`; 51 `print()` (users.py 27, bootstrap.py 17, crud.py 6); 22 `db.commit()` in the api layer across 8 files; 110/253 endpoints without `response_model`.

**Safety-net note:** Tasks 202, 204, 206, 209 touch authorization paths. The integration suite (issue #106) is still being scaffolded — land a permission-matrix test (role student/tutor/lecturer/admin × endpoint) for the submissions and tests routers BEFORE starting 202 and 204. The rest are safe with route-level smoke tests plus OpenAPI-schema diffing (`GET /openapi.json` before/after).

## TASK-209: Unify 404-vs-403 semantics between generated and hand-written routes

- **Category:** consistency / api-design — **Priority: P2** — **Effort: M**
- **Files:** `business_logic/crud.py` (84, 89, 169, 293, 298 — permission denial ⇒ `NotFoundException`, including create with no handler), vs hand-written routers raising `ForbiddenException` for the same situation (122 `ForbiddenException` vs 124 `NotFoundException` raises across api/); e.g. `api/examples.py:470` (403 for create) vs `crud.py:84` (404 for create).

**Problem:** For CrudRouter entities an unauthorized user gets 404 (resource-hiding), but for submissions/examples/messages the same failure returns 403 with a descriptive message. Clients can't rely on either convention; resource-existence leaks are inconsistent (the 403 paths confirm existence).

**Steps:**
1. Decide the policy per sensitivity class (suggest: 404 for object-scoped reads where existence is sensitive; 403 for action-denied on resources the caller can already see); write it into `exceptions/exceptions.py`'s conventions docstring.
2. Add a `PermissionDeniedAsNotFound` helper or a `hide_existence=True` flag on `ForbiddenException` so intent is explicit at call sites.
3. Sweep the TASK-202 permission-ladder sites while extracting that helper (one place to set the policy), then sweep `api/examples.py`'s nine `permitted(...)` guards.

**Risks / verification:** Status-code changes are breaking for API clients (web, VS Code extension) — coordinate; cover with the permission-matrix tests from TASK-202.
