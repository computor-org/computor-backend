# Development

Setup, the daily cycle, and the recurring tasks: adding an entity, migrations, code
generation, and tests.

## Prerequisites

- **Python 3.10+**, **Docker + Docker Compose**, **Git**
- Recommended: VS Code / PyCharm, an API client (HTTPie/Postman), a DB client (DBeaver)

## Setup

```bash
# 1. Virtualenv + editable installs (order matters: types → client → cli → backend)
python3.10 -m venv .venv && source .venv/bin/activate
pip install -e computor-types/ -e computor-client/ -e computor-cli/ -e computor-backend/

# 2. Environment — copy the template to the repo root, then fill in secrets
cp ops/environments/.env.common.template .env

# 3. Docker services (postgres, redis, temporal + UI, minio, traefik, workers)
./computor.sh up dev -d

# 4. Backend on :8000 — runs Alembic migrations and seeds the admin user on start
bash api.sh

# 5. Frontend on :3000 (separate terminal)
bash web.sh
```

**Editable installs** mean code changes take effect without reinstalling. **Never** use
`docker compose` directly — `./computor.sh` (via `ops/lib/common.sh`) picks the right
compose overlays from the `dev|prod` argument and the `.env` flags (`CODER_ENABLED`,
`KEYCLOAK_ENABLED`, `GIT_SERVER`, `MATLAB_ENABLED`, `UPDATE_ENABLED`); the old
`startup.sh`/`stop.sh`/`maintenance.sh` still work as wrappers. `api.sh` applies
migrations automatically, so `migrations.sh` is only for the rare standalone case.

Verify:

```bash
computor --help          # CLI installed
computor login           # authenticate against the API
curl localhost:8000/docs # Swagger UI is up
```

## Daily cycle

```bash
source .venv/bin/activate
git pull                       # on the active release branch
./computor.sh up dev -d        # add --build to rebuild worker/images after code changes
bash api.sh                    # terminal 1
bash web.sh                    # terminal 2
git checkout -b feat/your-thing
# …work…  →  bash test.sh  →  git commit  →  push  →  PR into the release branch
./computor.sh down             # stop docker services (auto-detects dev/prod)
```

> Worker code is **baked into the worker images**. After editing anything under `tasks/`,
> rebuild with `./computor.sh up dev --build -d` or the workers keep running stale code.

## Adding an entity, end-to-end

The full loop for a new entity (e.g. `Assignment`). Each step lives in the package named.

1. **DTOs** — `computor-types/…/assignments.py`: an `AssignmentInterface(EntityInterface)`
   plus `AssignmentCreate/Get/List/Update/Query`. See
   [backend-patterns.md](backend-patterns.md#entityinterface--dtos).
2. **Model** — `computor-backend/…/model/assignment.py`: a SQLAlchemy model inheriting the
   audit base; add the reverse `relationship` on related models.
3. **Migration** — `alembic revision --autogenerate -m "add assignments"`, review it, then
   `alembic upgrade head`. See [Migrations](#migrations).
4. **Business logic** — `business_logic/assignments.py`: functions taking
   `(data, permissions, db)` that check permissions and return DTOs.
5. **API** — `api/assignments.py`: thin endpoints delegating to business logic; register the
   router in `server.py`.
6. **Permissions** — register a handler for the model in `permissions/` if it needs
   entity-specific rules.
7. **Regenerate** — `bash generate.sh all` (TS types, clients, python-client, schemas).
8. **Test** — add `tests/test_assignments.py`; run `bash test.sh`.

## Migrations

Alembic derives migrations from the SQLAlchemy models. Migration files live in
`computor-backend/src/computor_backend/alembic/versions/`.

```bash
alembic revision --autogenerate -m "message"   # generate from model changes
alembic revision -m "message"                   # empty (for data migrations)
alembic upgrade head        # apply all pending
alembic downgrade -1        # roll back one
alembic current / history   # inspect state
```

Guidelines:

- **Always review autogenerated migrations** and implement `downgrade()`.
- **Add columns nullable first**, backfill in a separate step, then set `NOT NULL` — avoids
  breaking a live schema.
- Migration headers vary between `revision = "..."` and `revision: str = "..."`; use
  `alembic heads` (not grep) to find the correct parent when hand-authoring one.
- To reset a dev DB: `./computor.sh down` intentionally disables `-v`; remove the postgres
  data dir under `${SYSTEM_DEPLOYMENT_PATH}/postgres` manually, then `./computor.sh up dev -d`
  + `api.sh`.

## Code generation

`generate.sh` regenerates web/client artifacts offline from the Pydantic models.

```bash
bash generate.sh types          # TypeScript interfaces
bash generate.sh clients        # TypeScript API clients
bash generate.sh python-client  # computor-client (needs the backend importable)
bash generate.sh schemas        # JSON schemas for meta.yaml / test.yaml
bash generate.sh all            # everything (default)
```

The checked-in `computor-web/src/generated/` can be **stale**, so a full regen may bundle
unrelated drift — for a focused change, edit the generated file surgically and validate
with `npx tsc --noEmit`. `computor-web` uses **yarn** (never `npm install`).

## Tests

```bash
bash test.sh                              # all tests
pytest path/to/test_x.py::test_fn         # a single test
pytest -m unit  /  -m integration         # by marker
pytest --cov=computor_backend             # coverage
```

Write unit tests against `business_logic/` functions and integration tests against the API
via `TestClient`. Common fixtures (`client`, `db`, `admin_token`, `test_course`) live in
`tests/conftest.py`.

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| Port already in use | `lsof -i :8000` → `kill -9 <PID>` |
| Import errors for `computor_*` | Re-run the editable installs (order: types → client → cli → backend) |
| Worker runs old code | Rebuild: `./computor.sh up dev --build -d` (source is baked into worker images) |
| DB connection fails | `docker ps | grep postgres`; check `.env` datastore hosts (dev = `localhost`) |
| Migration "table already exists" / out of sync | `alembic current`; `alembic stamp head`; or reset the dev DB (above) |
